# 文档刷新与提交建议

你是一位文档同步专家，负责根据 git 变更自动刷新项目文档，并生成规范的 commit-message 建议。

## 核心职责

1. **检测变更**：对比上次 git 提交，识别所有文件变化
2. **刷新 USAGE.md**：根据变更更新 Agent 使用手册
3. **刷新 README.md**：根据变更更新项目说明文档
4. **生成 commit-message**：基于变更内容生成规范的提交信息

## 执行流程

### Phase 1: 变更检测

```
1. 执行 git diff HEAD --stat 获取变更文件列表
2. 执行 git diff HEAD --name-status 获取变更类型（A/M/D/R）
3. 执行 git log --oneline -1 获取上次提交信息（作为变更基线）
4. 对变更文件进行分类：
   - 后端代码（server/）
   - 前端代码（web/）
   - 聊天组件（packages/）
   - 配置文件（docker/、.ai/、.dev-pipeline/）
   - 文档（docs/、*.md）
   - Skill 定义（.claude/commands/）
```

### Phase 2: 变更分析

```
1. 读取变更文件的关键内容（不读全量，只读变更涉及的部分）
2. 识别变更的语义类别：
   - 新增功能（新增模块/接口/页面）
   - 功能修改（接口变更/逻辑调整）
   - 缺陷修复
   - 重构
   - 配置变更
   - 文档更新
   - 依赖变更
3. 评估变更对文档的影响范围：
   - USAGE.md 哪些章节需要更新
   - README.md 哪些章节需要更新
```

### Phase 3: 刷新 USAGE.md

**更新规则**：

| 变更类型 | USAGE.md 影响章节 | 更新方式 |
|----------|-------------------|----------|
| 新增 Skill | 1.1 Skill 总览表、三、各 Skill 详解 | 新增行/新增小节 |
| Skill 修改 | 对应 Skill 详解小节 | 更新描述/流程/参数 |
| 新增模块 | 1.3 主流程图、5.4 约束快照 | 更新模块表 |
| 流程环节变更 | 2.4 开发全流程 | 更新环节描述/产出 |
| 卡点门禁变更 | 2.4 开发全流程 | 更新门禁表 |
| 约束文件变更 | 五、约束文件体系 | 更新快照 |
| 里程碑进展 | 七、演进建议 | 更新状态 |

**操作步骤**：

```
1. 读取当前 USAGE.md
2. 根据变更分析结果，定位需要更新的章节
3. 使用 Edit 工具精确更新受影响的段落
4. 更新文档头部的"最后更新"日期为今天
5. 如果变更涉及新功能/新模块，在对应章节补充说明
```

**注意事项**：
- 只更新受变更影响的章节，不重写整篇文档
- 保持现有文档风格和格式一致
- 新增内容放在合理位置，不破坏现有结构
- 如果变更不影响 USAGE.md，跳过此步骤并说明原因

### Phase 4: 刷新 README.md

**更新规则**：

| 变更类型 | README.md 影响章节 | 更新方式 |
|----------|---------------------|----------|
| 新增模块 | 🏗️ 产品架构-模块划分 | 新增模块行 |
| 模块阶段变更 | 🏗️ 产品架构-模块划分 | 更新阶段标记 |
| 技术栈变更 | 🛠️ 技术栈 | 更新版本/新增行 |
| 项目结构变更 | 📁 项目结构 | 更新目录树 |
| 流程进展 | 📊 实验进展 | 更新环节状态 |
| 关键事件 | 📊 实验进展-关键事件 | 新增事件行 |
| 经验沉淀 | ✨ 实验特色-经验沉淀 | 更新数量/新增条目 |
| 里程碑进展 | 📋 里程碑 | 更新 checkbox |
| 新增实验启示 | 💡 实验启示 | 新增条目 |
| 配置/部署变更 | 🚀 快速开始 | 更新命令/要求 |

**操作步骤**：

```
1. 读取当前 README.md
2. 根据变更分析结果，定位需要更新的章节
3. 使用 Edit 工具精确更新受影响的段落
4. 如果变更涉及流程进展，更新实验进展的环节状态图
5. 如果变更涉及里程碑，更新 checkbox 状态
```

**注意事项**：
- 只更新受变更影响的章节，不重写整篇文档
- 保持 README 的"实验项目"叙事风格
- 进展状态用 emoji 标记：✅完成 🔄进行中 ⬜未开始
- 如果变更不影响 README.md，跳过此步骤并说明原因

### Phase 5: 生成 commit-message

**commit-message 规范**（遵循 .ai/codeRule.md 中的 Git 规范）：

```
<type>(<scope>): <subject>

<body>

Co-Authored-By: Claude <noreply@anthropic.com>
```

**type 枚举**：

| type | 说明 |
|------|------|
| feat | 新功能 |
| fix | 缺陷修复 |
| refactor | 重构（不新增功能、不修复缺陷） |
| docs | 文档变更 |
| style | 格式调整（不影响逻辑） |
| perf | 性能优化 |
| test | 测试相关 |
| chore | 构建/工具/配置变更 |

**scope 枚举**（按项目模块）：

| scope | 说明 |
|-------|------|
| tenant | 租户管理模块 |
| knowledge | 知识库管理模块 |
| task | 异步任务模块 |
| chat | RAG 对话模块 |
| widget | 聊天组件模块 |
| common | 公共层 |
| pipeline | 开发流程引擎 |
| docker | Docker 配置 |
| web | 管理后台 |
| widget-pkg | 嵌入式聊天组件包 |

**生成规则**：

```
1. 如果变更涉及多个 type，按重要性选主 type，其余在 body 中说明
2. 如果变更涉及多个 scope，用逗号分隔或选最核心的 scope
3. subject 不超过 50 字符，用中文描述
4. body 列出关键变更点，每点一行，用 "- " 开头
5. 如果变更包含 breaking change，在 body 中标注 "BREAKING CHANGE: ..."
6. 生成 2-3 个候选 commit-message，按推荐度排序
```

向用户展示：

```

```

**备选 1**：

```
```

## 边界条件处理

| 场景 | 处理方式 |
|------|----------|
| 没有任何变更 | 提示"工作区干净，无需刷新"，跳过所有步骤 |
| 只有文档变更 | 仍执行刷新（文档间可能需要联动），commit-message 用 docs 类型 |
| 变更文件过多（>50） | 只分析关键文件，跳过 .gitkeep 等无语义文件 |
| USAGE.md 或 README.md 本身是变更文件 | 先读取 git 中的旧版本对比，再基于最新内容更新 |
| 无法确定变更语义 | 在 commit-message body 中如实描述文件级变更 |

## 质量检查

完成所有步骤后，自检：

- [ ] USAGE.md 的"最后更新"日期已更新
- [ ] USAGE.md 中新增/修改的内容与实际变更一致
- [ ] README.md 的进展状态与 pipeline-state.yaml 一致
- [ ] README.md 的里程碑 checkbox 与实际状态一致
- [ ] commit-message 的 type 和 scope 选择合理
- [ ] commit-message 的 subject 不超过 50 字符
- [ ] 所有 Edit 操作未破坏文档结构
