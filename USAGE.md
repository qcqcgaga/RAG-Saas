## 一、全局认知

### 1.1 项目 Skill 总览

本项目在 `.claude/commands/` 下维护 9 个自定义 Skill，覆盖从产品定义到复盘优化的完整闭环：

| Skill | 命令 | 定位 | 类型 |
|-------|------|------|------|
| 产品设计师 | `/product` | 产品定位与功能边界定义 | 主流程 |
| 架构师 | `/architect` | 技术选型与架构设计 | 主流程 |
| 开发全流程引擎 | `/dev-pipeline` | 需求→设计→编码→测试→部署全流程，强制前置经验教训 | 主流程 |
| 迭代复盘与优化 | `/retrospect` | 版本复盘、约束审视、技术债治理、质量趋势、Skill优化 | 主流程 |
| 文档体系构建 | `/build_doc_sys` | 贯穿全生命周期的 LLM 友好文档体系 | 跨阶段支撑 |
| 文档刷新与提交建议 | `/refresh-docs` | 根据git变更刷新USAGE.md/README.md，生成commit-message | 跨阶段支撑 |
| Skill 编写专家 | `/create_skill` | 创建/编辑/验证自定义 Skill | 专项工具 |
| Maven 包管理专家 | `/maven-expert` | Maven 依赖诊断与修复 | 专项工具 |
| 重构实施专家 | `/refactor-expert` | 识别架构偏差，执行零风险增量重构 | 专项工具 |

### 1.2 核心理念

```
约束驱动：所有 Skill 的输出最终沉淀为 .ai/ 约束文件，后续开发必须遵守
人机协作：AI 生成方案/代码，人做决策和评审
渐进细化：从宏观到微观，每步确认后再推进
状态持久：流程状态写入文件，可断点续跑
闭环改进：交付后复盘，复盘结果回灌约束文件和 Skill 体系
```

## 二、主流程：产品 → 架构 → 开发 → 复盘

### 2.1 流程总览

```
阶段1          阶段2            阶段3              阶段4
/product  →  /architect  →  /dev-pipeline  →  /retrospect
(产品定义)    (架构设计)      (开发全流程)       (复盘优化)
```

每个阶段的产出是下一阶段的输入，**严格串行**，不可跳过。
阶段 4（复盘）的产出回灌到阶段 1-2 的约束文件，形成**持续改进闭环**。

---

### 2.2 阶段一：产品定义 `/product`

**何时使用**：项目刚启动，或需要重新审视产品方向时。

**核心产出**：`.ai/product.md`

**执行流程**：

```
Phase 1: 现状感知
  └─ 读取现有 .ai/ 文件，判断 product.md 填写状态

Phase 2: 产品定位
  ├─ 2.1 一句话描述（用户 + 场景 + 价值）
  ├─ 2.2 目标用户（角色、规模、技术水平、场景）
  └─ 2.3 核心价值（独特价值、可量化/可感知）

Phase 3: 功能定义
  ├─ 3.1 功能发现（用户故事驱动 + 场景推演 + MVP 思维）
  ├─ 3.2 功能优先级（P0/P1/P2）
  └─ 3.3 功能描述规范（具体到可验证）

Phase 4: 功能边界
  ├─ 4.1 不做列表（明确排除 + 原因）
  ├─ 4.2 非功能性要求（性能/安全/可用性/数据量）
  └─ 4.3 里程碑规划（MVP → V1 → V2）

Phase 5: 约束联动与输出
  ├─ 5.1 联动检查（与 tech/structure/codeRule 的一致性）
  ├─ 5.2 输出到 .ai/product.md
  └─ 5.3 同步 CLAUDE.md 状态
```

**使用示例**：

```
> /product

（Agent 读取现有文件后，逐层提问引导）

Agent: "这个项目要解决什么问题？谁会使用它？"
你:   "面向独立开发者，上传文档就能生成聊天组件"

Agent: "用一句话描述这个产品，你会怎么说？"
你:   "DocChat 是面向独立开发者的文档智能客服 SaaS，上传文档5分钟上线聊天组件"

...（逐步收敛）

Agent: 输出 .ai/product.md，提示运行 /architect
```

**关键交互规则**：
- Agent 每次只问 1-2 个问题，避免信息过载
- 你可以随时要求回退到之前的决策点
- 每个"不做"项需要说明原因
- 功能描述必须具体到可验证（❌ "用户相关功能" → ✅ "用户的增删改查 + 角色分配"）

### 2.3 阶段二：架构设计 `/architect`

**何时使用**：产品定义完成后（`product.md` 已填写），或技术栈需要调整时。

**前置条件**：`.ai/product.md` 已完成。

**核心产出**：`.ai/structure.md`、`.ai/tech.md`、`.ai/codeRule.md`（三文件联动更新）

**执行流程**：

```
Phase 1: 项目理解
  └─ 读取所有 .ai/ 文件，确认产品需求和技术约束

Phase 2: 技术选型
  ├─ 后端选型（语言/框架/ORM/数据库/认证/API风格）
  ├─ 前端选型（框架/UI库/状态管理/构建工具）
  └─ 工具链选型（格式化/Lint/测试/容器化）
  └─ 每项选型必须说明理由和权衡

Phase 3: 架构设计
  ├─ 3.1 目录结构设计
  ├─ 3.2 分层架构设计（Controller → Service → Repository → Entity）
  └─ 3.3 模块设计（划分/通信/共享/边界）

Phase 4: 质量与安全
  ├─ 质量属性（可测试性/可维护性/可观测性/性能/可扩展性）
  └─ 安全设计（认证授权/数据安全/输入安全/通信安全/依赖安全/运维安全）

Phase 5: 约束同步与验证
  ├─ 同步检查三个约束文件一致性
  ├─ 完整性检查
  └─ 输出变更摘要
```

**使用示例**：

```
> /architect

Agent: "我已读取 product.md，DocChat 定位是文档智能客服 SaaS。先确认几个信息：
       - 项目类型？全栈应用
       - 预期数据量和并发量？单租户 < 100 文档，对话 < 100 QPS
       - 团队技术背景？Java + Vue
       ..."

Agent: "后端语言选型，有两个候选：
       1. Java 21 — 团队熟悉、生态成熟、虚拟线程支持
       2. Python — AI 生态直连便利
       你倾向哪个？"
你:   "Java 21"

...（逐步收敛）

Agent: 输出 .ai/tech.md + .ai/structure.md + .ai/codeRule.md
```

**注意事项**：
- 技术选型每项都有 2-3 个候选方案和优劣分析，**不是直接给答案**
- 安全和质量要求根据项目实际风险等级调整，不盲目追求最高标准
- 架构设计允许 MVP 阶段适度简化

### 2.4 阶段三：开发全流程 `/dev-pipeline`

**何时使用**：产品和架构定义完成后，准备进入实质性开发时。

**前置条件**：`.ai/product.md` + `.ai/tech.md` + `.ai/structure.md` + `.ai/codeRule.md` 已完成。

**核心产出**：`docs/pipeline/{run_id}/` 下的全流程文档 + 代码

**14 个流程环节 + 7 个卡点门禁**：

> 💡 **V1 增强**：流程启动前强制读取 `tools-and-lessons.yaml` 中所有 critical 经验教训，向用户展示摘要。人工测试环节新增完整重建三步法（遵循 L-DEPLOY-006/007），每次修复代码后也必须重建+重启。

```
[1] 需求分析
     │
[2] 需求评审 ★ G1 卡点
     │
[3] 技术方案设计 ─── 会调用 /architect 的产出
     │
[4] 技术方案评审 ★ G2 卡点
     │
[5] 详细设计
     │
[6] 详细设计评审 ★ G3 卡点
     │
[7] 编码实现
     │
[8] 代码评审 ★ G4 卡点
     │
[9] 单元测试
     │
[10] 集成测试
     │
[11] 人工测试 ─── 测试前必须调用 /user-manual 刷新用户手册
     │
[12] 测试评审 ★ G5 卡点
     │
[13] 部署上线
     │
[14] 上线验证 ★ G6 卡点 → 生成流程内复盘报告
```

> 💡 **人工测试环节**是 MVP 部署复盘后新增的环节。自动化测试全绿≠系统可用，人工测试以用户故事为单位验证端到端流程，是自动化测试的必要补充。

**关键机制**：

| 机制 | 说明 |
|------|------|
| **用户是监督者** | AI 生成文档/代码，每个环节产出供你评审，通过后才推进 |
| **卡点不可跳过** | 7 个门禁必须通过评审才能继续 |
| **断点续跑** | 状态写入 `.dev-pipeline/pipeline-state.yaml`，中断后可恢复 |
| **多 run 隔离** | 每次执行有独立 run_id（如 `mvp-2026-06-24`），产出互不干扰 |
| **可回退** | 评审不通过时回退到指定环节，不是失败是质量保障 |

**使用示例**：

```
> /dev-pipeline

Agent: "流程尚未开始。请指定本次执行的 scope（如 mvp、v1、hotfix-xxx）："
你:   "mvp"

Agent: "run_id 确认为 mvp-2026-06-24。开始环节1：需求分析。"

...（Agent 生成 PRD 文档）

Agent: "以上是【需求分析】的产出，请审阅。是否满意？"
你:   ✅ 满意，继续

Agent: "进入环节2：需求评审（G1 卡点）。"
Agent: 展示检查清单，逐项确认...

...（逐环节推进）
```

**命令参数**：

| 参数 | 说明 |
|------|------|
| `resume` | 从上次中断的位置继续执行 |
| `phase=环节名` | 跳转到指定环节（如 `phase=coding`） |
| `status` | 仅展示当前流程状态，不执行 |

**产出文档清单**（每次 run）：

### 2.5 阶段四：迭代复盘与优化 `/retrospect`

**何时使用**：
- 版本交付后进行深度复盘
- 开发流程 G6 卡点通过后（自动提示衔接）
- 需要审视约束文件有效性时
- 需要治理技术债务时
- 需要分析质量趋势时

**前置条件**：至少完成一个 `/dev-pipeline` run。

**核心产出**：`docs/pipeline/{run_id}/retrospective-deep.md` + `tech-debt-plan.md` + `constraint-changes.md` + `skill-optimization.md`

**5 个复盘维度**：

```
Phase 1: 版本交付复盘
  ├─ 1.1 交付质量评估（缺陷分布、测试通过率、覆盖率）
  ├─ 1.2 交付效率评估（总周期、环节耗时、返工次数）
  └─ 1.3 交付满意度（交互式问答）

Phase 2: 约束文件审视
  ├─ 2.1 product.md 审视（功能边界、优先级、非功能性要求）
  ├─ 2.2 tech.md 审视（选型合适性、版本过时、技术空白）
  ├─ 2.3 structure.md 审视（模块划分、依赖关系、目录结构）
  └─ 2.4 codeRule.md 审视（命名规范、复杂度、安全规则）

Phase 3: 技术债务治理
  ├─ 3.1 债务识别（coding-log + code-review + TODO/FIXME扫描）
  ├─ 3.2 债务分类（架构/代码质量/安全/测试/性能）
  ├─ 3.3 债务评估（影响范围 + 修复成本 + 风险等级 + 紧迫程度）
  └─ 3.4 治理计划（紧急/一般/持续 三级）

Phase 4: 质量趋势分析
  ├─ 4.1 数据收集（多 run 关键指标）
  ├─ 4.2 趋势分析（MVP → V1 → V2 指标对比）
  └─ 4.3 趋势预警（恶化/波动/持续低位）

Phase 5: Skill 体系优化
  ├─ 5.1 使用反馈收集（体验/繁琐/空白/协作/冲突）
  ├─ 5.2 优化建议（每个 Skill 的改进项和优先级）
  └─ 5.3 缺口识别（需要新建的 Skill）
```

**闭环机制**：复盘产出的约束变更建议，引导用户通过对应 Skill 执行（如 `/product` 调整 product.md，`/architect` 调整约束文件），形成持续改进闭环。

**命令参数**：

| 参数 | 说明 |
|------|------|
| `run_id=xxx` | 指定复盘的流程执行 ID，默认 latest |
| `dimension=all/quality/process/tech-debt/constraints/skill` | 复盘维度，默认全维度 |
| `depth=quick/deep` | 快扫（30分钟）或深挖（1小时+），默认深挖 |

### 2.6 跨阶段支撑：文档体系构建 `/build_doc_sys`

> 💡 **定位调整**：`/build_doc_sys` 不是主流程中的一个阶段，而是**贯穿项目全生命周期的支撑能力**。

**介入时机**：

| 时机 | 模式 | 目的 |
|------|------|------|
| 架构设计完成后、开发前 | 新项目模式 | 为开发阶段准备 LLM 友好的上下文 |
| 开发过程中（按需） | 文档模式（增量） | 新增模块的文档补充 |
| 版本交付后 | 文档模式（全量/增量） | 同步代码变更到文档体系 |
| 复盘优化后 | 文档模式（增量） | 落实复盘中的约束变更到文档 |

**两种模式**：

#### 新项目模式（5 个阶段）

```
[1] PRD 解析 → [2] 架构设计 → [3] 需求拆解 → [4] 生成 CLAUDE.md 开发指南 → [5] 模块开发文档
```

适用场景：项目刚启动，从 PRD 出发生成架构设计、需求拆解、开发规范。

> 💡 **与主流程的关系**：如果已通过 `/product` + `/architect` 完成了产品定义和架构设计，
> 新项目模式的阶段 1-2 可以跳过，直接从阶段 3（需求拆解）开始。

#### 文档模式（8 个阶段）

```
[1] 项目探查 → [2] 架构分类 → [3] 编写 CLAUDE.md → [4] 基础模块文档
→ [5] 业务模块文档 → [6] 配置层文档 → [7] 变更日志 → [8] 交叉验证
```

适用场景：项目已有代码，需要梳理生成 LLM 友好的分层文档体系。

**文档分层架构**：

```
.claude/CLAUDE.md                    = 主索引（项目地图，始终加载到上下文）
.claude/docs/{模块}/README.md        = 模块级文档（按需加载）
.claude/docs/{模块}/*.md             = 深度参考文档（API详情、数据模型、坑点）
.claude/docs/{模块}/CHANGELOG.md     = 模块变更历史（调试和回滚参考）
```

**关键规则**：
- `.claude/CLAUDE.md` 控制在 150 行以内
- 模块 README.md 控制在 200 行以内
- 多模块时逐个串行处理，防止上下文溢出
- 支持增量更新（`.claude/docs/_progress.md` 记录断点）

## 三、各 Skill 详解

### 3.1 `/product` — 产品设计师

| 属性 | 说明 |
|------|------|
| 输入 | 用户对产品的想法（可模糊） |
| 输出 | `.ai/product.md` |
| 交互方式 | 逐层提问引导，每步确认 |
| 下游 Skill | `/architect` |

**核心价值**：防止功能蔓延。产品定义阶段明确"做什么"和"不做什么"，比后续任何阶段都重要。

**产出字段清单**：

| 字段 | 必填 | 说明 |
|------|------|------|
| 产品定位 | ✅ | 一句话描述 + 目标用户 + 核心价值 |
| 角色定义 | ✅ | 系统角色表 |
| 核心功能 | ✅ | 功能模块表（模块名 + 描述 + 优先级 + 状态） |
| 功能边界（不做） | ✅ | 明确列出不做的事项及原因 |
| 非功能性要求 | ✅ | 性能/安全/可用性/数据量/可测试性/可观测性 |
| 里程碑 | ✅ | MVP/V1/V2 的目标和时间 |
| 变更记录 | ✅ | 每次变更的日期和内容 |

---

### 3.2 `/architect` — 架构师

| 属性 | 说明 |
|------|------|
| 输入 | `.ai/product.md`（必须已有） |
| 输出 | `.ai/structure.md` + `.ai/tech.md` + `.ai/codeRule.md` |
| 交互方式 | 逐项选型引导，每项说明理由和权衡 |
| 上游 Skill | `/product` |
| 下游 Skill | `/dev-pipeline` |

**核心价值**：约束驱动。技术选型和架构规则写入约束文件，后续所有开发（包括 AI 生成代码）必须遵守。

**约束文件优先级**：

```
.ai/codeRule.md     >  代码层面硬性规则，不可违反
.ai/structure.md    >  架构层面组织规则，新增文件必须遵循
.ai/tech.md         >  技术选型约束，不得引入未列出技术
.ai/product.md      >  功能边界，不得实现"不做"列表中的功能
```

---

### 3.3 `/dev-pipeline` — 开发全流程引擎

| 属性 | 说明 |
|------|------|
| 输入 | `.ai/` 约束文件（全部已有）+ 用户原始需求 |
| 输出 | `docs/pipeline/{run_id}/` 全流程文档 + 代码 |
| 交互方式 | 环节产出评审，卡点逐项确认 |
| 上游 Skill | `/product` + `/architect` |
| 内嵌协作 | `/code-review`（代码评审环节）+ `/verify`（部署验证环节）+ `/user-manual`（人工测试环节） |

**元数据文件**：

| 文件 | 用途 |
|------|------|
| `.dev-pipeline/phases.yaml` | 14 个环节定义（依赖关系、产出物） |
| `.dev-pipeline/gates.yaml` | 7 个卡点定义（检查清单、通过条件） |
| `.dev-pipeline/tools-and-lessons.yaml` | 工具选择和经验记录 |
| `.dev-pipeline/pipeline-state.yaml` | 流程状态跟踪（当前环节、评审历史） |
| `.dev-pipeline/templates/` | 各环节文档模板 |

---

### 3.4 `/retrospect` — 迭代复盘与优化

| 属性 | 说明 |
|------|------|
| 输入 | `/dev-pipeline` 产出文档 + `.ai/` 约束文件 + git 历史 |
| 输出 | `retrospective-deep.md` + `tech-debt-plan.md` + `constraint-changes.md` + `skill-optimization.md` |
| 交互方式 | 数据展示 + 交互式问答 |
| 上游 Skill | `/dev-pipeline` |
| 下游回灌 | `/product`、`/architect`（约束变更）、`/create_skill`（Skill 优化） |

**核心价值改进。复盘不是终点，而是下一轮迭代的起点——产出回灌约束文件和 Skill 体系，形成持续改进闭环。

**5 个维度**：版本交付复盘 → 约束文件审视 → 技术债务治理 → 质量趋势分析 → Skill 体系优化

---

### 3.5 `/build_doc_sys` — 文档体系构建（跨阶段）

| 属性 | 说明 |
|------|------|
| 输入 | 项目源码（文档模式）或 PRD（新项目模式） |
| 输出 | `.claude/CLAUDE.md` + `.claude/docs/` 分层文档 |
| 交互方式 | 阶段产出确认 |
| 自动检测 | 根据项目状态自动选择文档模式或新项目模式 |

**上下文管理关键规则**：
- 多模块逐个串行处理，禁止一次性处理所有模块
- 每完成一个模块 → 更新 `_progress.md` → 不再引用该模块源码
- 处理 3+ 个大模块后 → 主动提示开新会话继续
- 全部完成后删除 `_progress.md`

## 四、专项工具 Skill

### 4.1 `/maven-expert` — Maven 包管理专家

**何时使用**：
- Maven 依赖冲突（`NoSuchMethodError`、`ClassNotFoundException`）
- 版本合规检查（对照 `.ai/tech.md` 约束）
- 安全漏洞检测（OWASP 审计）
- 构建配置优化
- `javax` → `jakarta` 迁移

**执行流程**：

```
Phase 1: 项目结构分析（读取 pom.xml + .ai/ 约束文件）
Phase 2: 依赖健康检查（冲突/缺失/版本合规/安全漏洞）
Phase 3: 构建配置检查（POM 结构/插件配置/构建性能）
Phase 4: 问题修复（依赖冲突/缺失/安全漏洞）
Phase 5: 验证与报告（编译/测试/依赖树/安全审计）
```

**输出**：Maven 依赖健康报告（问题汇总 + 修复建议 + 技术栈合规性）

**自动对照约束**：自动读取 `.ai/tech.md` 和 `.ai/codeRule.md`，检查依赖版本是否符合约定。

---

### 4.2 `/create_skill` — Skill 编写专家

**何时使用**：
- 需要创建新的自定义 Skill
- 修改现有 Skill
- 验证 Skill 格式是否规范
- 设计 Skill 间的协作关系
- 配置 Skill 按需加载规则

**支持两种 Skill 格式**：

| 格式 | 适用场景 |
|------|----------|
| YAML | 技术实现类、协作定义类 Skill |
| Markdown | 流程分析类 Skill（如本项目的 /product、/architect） |

**按需加载规则**：根据研发类型（Java/Python/Go/Vue/React/Qt）和部门类型（产品/设计/测试/运维/安全/数据/项目管理）自动加载对应的规则模块。

---

### 4.3 `/refactor-expert` — 重构实施专家

**何时使用**：
- 开发流程完成后，检查项目实际结构与 `.ai/structure.md` 规范的偏差
- 用户说"重构"、"对齐架构"、"整理结构"、"符合规范"
- 需要确保项目目录结构和代码组织符合架构规范

**核心原则**：
- **零删除** — 不删除任何已有文件
- **零破坏** — 不影响任何现有功能的运行
- **零中断** — 不影响现有 Skill 的执行逻辑
- **硬验收** — 项目可启动 + 用户检查功能全部通过

**执行流程**：

```
Phase 0: 环境准备
  ├─ 读取 .ai/ 规范文件（codeRule → structure → tech → product）
  └─ git status 记录当前状态

Phase 1: 差异分析（只读，不修改）
  ├─ 全量扫描实际结构 vs structure.md
  ├─ 差异分级（🔴严重 / 🟡中等 / 🟢轻微）
  ├─ 制定重构方案（新增/移动/代理/文档）
  └─ 输出差异报告 → 用户确认

Phase 2: 增量重构执行
  ├─ 🟢 无风险项：新增缺失目录和 package-info.java
  ├─ 🟢 无风险项：创建占位文件和导航文档
  ├─ 🟡 中风险项：移动文件 + 更新引用
  ├─ 🟡 中风险项：处理计划外包（保留/迁移/代理）
  └─ 🔴 严重项：需用户逐项确认

Phase 3: 启动验证
  ├─ Maven 编译 + 前端构建 + 组件构建
  ├─ 后端启动验证（无 ERROR / Bean 注入正常）
  └─ 用户功能验证清单（逐项确认）

Phase 4: 收尾
  ├─ 更新导航文档
  └─ 生成重构完成报告
```

**建议使用时机**：在 `/dev-pipeline` 完整流程执行之后调用，用于兜底开发流程中未能做到架构一致的地方。

## 五、约束文件体系

### 5.1 `.ai/` 目录结构

```
.ai/
├── product.md      # 产品定位和功能边界
├── tech.md         # 技术栈约定（语言、框架、版本）
├── structure.md    # 项目目录结构和分层架构规范
└── codeRule.md     # 代码规范（命名、复杂度、安全规则）
```

### 5.2 约束文件的作用

这些文件是**项目的宪法**——所有 Skill 的输出、所有 AI 生成的代码、所有开发决策，都必须遵守这些约束。

| 文件 | 核心内容 | 由哪个 Skill 生成/更新 |
|------|----------|----------------------|
| `product.md` | 定位、功能、边界、非功能性要求、里程碑 | `/product` |
| `tech.md` | 后端/前端/工具链技术栈 + 版本约束 + 决策记录 | `/architect` |
| `structure.md` | 目录结构 + 分层架构 + 模块划分 + 依赖关系 | `/architect` |
| `codeRule.md` | 命名规范 + 复杂度限制 + 安全规则 + 错误处理 + 日志 + 测试 + Git 规范 | `/architect` |

### 5.3 约束变更流程

```
1. 约束文件变更需明确记录原因和日期
2. 技术栈变更需同步更新 tech.md + structure.md + codeRule.md
3. 功能增减需更新 product.md
4. 变更记录写在各文件的"变更记录"表中
```

### 5.4 当前约束快照（DocChat 项目）

**技术栈**：Java 21 + Spring Boot 3.3 + PostgreSQL 16 + Milvus 2.4 + Redis 7 + Vue 3 + Ant Design Vue 4 + TypeScript + Vite 6

**模块划分**（8 个业务模块）：

| 模块 | 阶段 | 状态 |
|------|------|------|
| module-tenant / module-knowledge / module-task / module-chat / module-widget | MVP | ✅ 已上线 |
| module-apikey / module-stat / module-eval | V1 | 🔄 开发中（人工测试阶段） |
