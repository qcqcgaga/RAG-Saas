# 产品需求文档 (PRD)

> 项目：DocChat — 文档智能客服 SaaS
> 版本：1.0 (MVP)
> 日期：2026-06-24
> 作者：产品团队

## 1. 需求背景

### 1.1 项目背景

独立开发者和小团队在发布产品后，经常面临用户反复询问相同问题的情况。传统方案是编写 FAQ 页面，但用户更倾向于直接对话获取答案。自建 RAG（检索增强生成）系统需要投入大量精力在向量数据库选型、文档切分策略、LLM 集成等技术细节上，对于小团队而言成本过高。

DocChat 旨在解决这一痛点：用户上传产品文档，系统自动切分、向量化，生成可嵌入任意网站的聊天组件，终端访客即可获得基于文档的智能问答服务。

### 1.2 项目目标

**业务目标**：
- MVP 上线后吸引 50+ 独立开发者注册试用
- 付费转化率 > 5%（V1 阶段验证）

**用户目标**：
- 5 分钟内完成"注册 → 上传文档 → 嵌入组件"全流程
- 对话回答准确率 > 80%（基于文档内容的检索命中率）
- 无需编写任何代码即可在网站中嵌入聊天组件

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| 租户 (Tenant) | 一个独立的工作空间，对应一个客户组织 |
| 知识库 (Knowledge Base) | 租户上传的文档集合，经过切分和向量化后用于 RAG 检索 |
| 文档切分 (Chunking) | 将长文档按策略拆分为小段，便于向量化检索 |
| 向量化 (Embedding) | 将文本段转化为高维向量，用于语义相似度检索 |
| RAG | Retrieval-Augmented Generation，检索增强生成，先检索相关文档片段再让 LLM 生成回答 |
| 聊天组件 (Widget) | 可嵌入第三方网站的 JS 聊天窗口，类似 Intercom/Drift |
| 评测集 (Eval Set) | 预设的问答对，用于自动评测检索和回答质量（V1 功能） |

## 2. 目标用户

### 2.1 用户角色

| 角色 | 描述 | 核心诉求 |
|------|------|----------|
| 租户管理员 | 注册账号的独立开发者或小团队负责人 | 快速上线文档客服，减少重复回答用户问题的负担 |
| 团队成员 | 被管理员邀请加入租户的协作者 | 管理知识库文档，配置聊天组件外观 |
| 终端访客 | 在嵌入的聊天组件中提问的最终用户 | 快速获得基于文档的准确回答 |

### 2.2 用户规模

- MVP 阶段目标：100+ 注册租户，日活 < 50
- 单租户并发访客 < 100
- 单租户文档数 < 100，单文档 < 50MB

## 5. 非功能性需求

| 维度 | 要求 | 验收标准 |
|------|------|----------|
| 性能 | 对话 API < 2s（P95），管理后台 API < 500ms（P95） | 压测验证：100 并发下 P95 延迟满足要求 |
| 可用性 | 单实例部署，不要求高可用 | Docker Compose 部署，单节点可运行 |
| 安全性 | 多租户严格隔离 + JWT + API Key + 输入校验 + 文件上传校验 | 1. 跨租户数据不可访问 2. 无硬编码密钥 3. 文件上传有类型+大小+文件头校验 |
| 数据量 | 单租户文档 < 100 个，单文档 < 50MB，对话并发 < 100 QPS | 数据模型设计支持上述容量 |
| 国际化 | 中文界面，不做 i18n | UI 文案全中文 |
| 可测试性 | 核心 Service 单元测试覆盖率 > 80% | CI 中验证覆盖率 |
| 可观测性 | SLF4J + Logback，Request-Id 链路追踪，结构化日志，日志脱敏 | 1. 每条日志包含 Request-Id 2. 敏感信息不入日志 |
| 部署 | Docker Compose（MVP） | 一键 `docker-compose up` 启动全部服务 |

## 6. 验收标准

### 6.1 MVP 验收标准

- [ ] 租户管理员可完成注册→登录→上传文档→查看文档列表→删除文档全流程
- [ ] 文档上传后自动触发切分+向量化任务，任务状态可查询
- [ ] 任务失败后可重试
- [ ] 终端访客可通过嵌入的聊天组件进行问答，2s 内开始响应
- [ ] 对话回答附带来源引用（文档名+段落）
- [ ] 对话即焚，不持久化到数据库
- [ ] 聊天组件可嵌入任意网页，支持品牌色、欢迎语、图标配置
- [ ] 多租户数据严格隔离，A 租户无法访问 B 租户数据
- [ ] Docker Compose 一键部署启动
- [ ] 核心 Service 单元测试覆盖率 > 80%

### 6.2 V1 验收标准

- [ ] API Key 可生成/吊销，每日调用次数受限
- [ ] 用量统计面板展示调用量、Token 消耗、对话数
- [ ] 评测集可管理问答对，自动评测检索 Hit Rate

## 7. 功能边界（不做）

| 不做项 | 原因 | 可能的V2规划 |
|--------|------|-------------|
| 实时计费/账单/支付集成 | MVP 只做调用次数硬限制，不做金额计算和支付 | V2 考虑 |
| 多语言 UI | 目标用户是独立开发者/小团队，中文界面即可 | V2 按需 |
| 对话历史持久化与导出 | 终端访客的对话是临时性的，问完即焚 | V2 按需 |
| 多知识库/多 Bot | 每个租户一个知识库一个 Bot，不做多实例 | V2 按需 |
| SSO/OAuth 第三方登录 | 目标用户是小团队，邮箱注册即可 | V2 考虑 |
| 文档协作编辑 | 产品定位是"上传+生成"，不是文档编辑器 | 不计划 |
| Webhook/事件回调 | MVP 无集成需求 | V2 按需 |
| 移动端适配（管理后台） | 管理后台桌面端优先，聊天组件本身响应式 | V2 按需 |

## 8. 依赖与约束

### 8.1 外部依赖

| 依赖项 | 类型 | 影响 |
|--------|------|------|
| 讯飞 Coding Plan API | 技术 | LLM 对话生成依赖此服务，服务不可用时对话功能不可用 |
| PostgreSQL 16+ | 技术 | 主数据存储，需保证数据库可用 |
| Milvus 2.4+ | 技术 | 向量检索依赖，服务不可用时对话检索不可用 |
| Redis 7.0+ | 技术 | 缓存+异步任务队列，服务不可用影响任务处理 |
| Docker + Docker Compose | 技术 | 部署依赖 |

### 8.2 约束条件

- 必须使用 Java 21 + Spring Boot 3.3+ + Vue 3 技术栈（见 `.ai/tech.md`）
- 代码必须遵循 `.ai/codeRule.md` 规范
- 项目结构必须遵循 `.ai/structure.md` 规范
- 单实例 Docker Compose 部署，不要求高可用
- 中文界面，不做国际化
