### 1.2 核心问题：纸面通过≠真实可用

**这是本次复盘最重要的发现。**

流程的6个卡点全部通过、127个测试用例100%通过、缺陷清单全部关闭——但实际部署后发现系统到处报错，经历了5个额外的阻塞问题才勉强跑起来。

| 问题 | 发现环节 | 本应在哪个环节发现 |
|------|---------|-------------------|
| Dockerfile context路径错误 | 部署上线 | 代码评审(G4) |
| JWT_SECRET默认值过短 | 部署上线 | 代码评审(G4) |
| logback prod不输出CONSOLE | 部署上线 | 代码评审(G4) |
| Flyway seed迁移冲突 | 部署上线 | 集成测试 |
| 前端tsconfig.node.json错误 | 部署上线 | 代码评审(G4) |

**根因**：
1. 集成测试使用H2内存数据库+MockBean，与真实PG/Milvus/Redis环境完全脱节
2. 缺少人工测试环节，没有人类在真实环境中端到端验证
3. G4虽然增加了"编译构建通过"和"应用启动验证"，但验证的是dev环境，不是prod环境

## 七、下次版本建议

1. **人工测试环节必须严格执行** — 这是MVP最大的教训，"自动化测试全绿"绝不等于"系统可用"
2. **集成测试应尽量在真实环境中执行** — H2+Mock只能验证逻辑，无法验证环境差异
3. **G4检查清单应增加Docker构建验证** — 不仅验证dev环境启动，还验证Docker构建和prod环境启动
4. **性能测试应有轻量级方案** — 不能总是"待压测"，至少应有基本的响应时间验证
5. **Embedding和LLM占位必须在V1替换** — 这两个占位使得核心功能（RAG对话）无法真正使用

## 八、改进后的流程对比

### 改进前

```
集成测试 → 测试评审(G5) → 部署上线 → 上线验证(G6)
              ↑                        ↑
        只看自动化结果              只验5个API
        H2全绿≠PG能跑             不验用户流程
```

### 改进后

```
集成测试 → 人工测试 → 人工测试评审(G5.5) → 测试评审(G5) → 部署上线 → 上线验证(G6)
              ↓              ↓                    ↓
        刷新用户手册    用户故事全通过?        含人工测试结果
        真实环境验证    问题修复→重启→重测     有真实数据支撑
        用户故事维度
```

**关键差异**：
1. 人工测试在**真实环境**中验证，弥补自动化测试的盲区
2. G5.5确保**用户故事级别**的端到端验证通过后才进入G5
3. 人工测试发现问题后**修复→重启→继续测试**，不走完整流程
4. 人工测试前**强制刷新用户手册**，确保测试引导与代码一致
