# 产品需求文档 (PRD)

> 项目：DocChat — 文档智能客服 SaaS
> 版本：2.0 (V1)
> 日期：2026-06-26
> 作者：产品团队
> 基线：基于 MVP（v1.0）增量迭代

## 1. 需求背景

### 1.1 项目背景

MVP 阶段已完成 DocChat 的核心功能闭环：租户管理 + 知识库管理 + 异步任务 + RAG 对话 + 聊天组件嵌入。MVP 验证了"上传文档→嵌入组件→智能问答"的核心价值链路。

但 MVP 阶段存在以下不足，限制了产品的商用能力：

1. **无 API Key 鉴权**：当前聊天组件仅用简单 token 标识租户，任何人拿到 token 就能无限调用，无法做访问控制和用量管理
2. **无用量统计**：管理员无法了解系统使用情况，不知道有多少对话、消耗了多少 Token，无法为后续付费提供数据支撑
3. **无质量评测**：RAG 检索质量无法量化评估，管理员不知道回答是否准确，无法有针对性地优化
4. **无法预览和模拟测试**：管理员配置聊天组件后无法在管理后台内预览效果，也无法模拟终端访客问答来验证 RAG 质量
5. **LLM 为 Mock 实现**：当前 LlmService 返回模拟文本，无法接入真实的 LLM API

### 1.2 项目目标

**业务目标**：
- V1 上线后，租户可通过 API Key 管理和用量统计实现"有限额的免费试用"，为付费转化铺路
- 通过评测集让 RAG 质量可量化、可迭代，建立用户信任
- 通过预览和模拟测试降低配置错误导致上线后体验差的风险

**用户目标**：
- 管理员可自助管理 API Key，控制调用额度，监控用量
- 管理员可创建评测集，自动评测检索质量，量化改进效果
- 管理员可在管理后台实时预览聊天组件外观并模拟问答
- 管理员可配置真实 LLM API 地址和密钥，获得真实 RAG 对话能力

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| API Key | 租户级别的访问密钥，用于聊天组件鉴权调用，每个租户可有多个 Key |
| 用量统计 (Usage Stats) | 按租户维度汇总的调用量、Token 消耗、对话数等统计数据 |
| 评测集 (Eval Set) | 预设的问答对集合，用于自动评测 RAG 检索和回答质量 |
| Hit Rate | 检索命中率，评测集中正确检索到相关文档的问答对占比 |
| 预览对话 (Preview Chat) | 管理员在管理后台内通过 iframe 发起的模拟对话，走与正式访客完全相同的后端代码路径，但不计入用量统计 |
| LLM API 配置 | 租户级别的 LLM 服务端点地址和密钥配置，用于替代系统默认的 LLM 服务 |

## 2. 目标用户

### 2.1 用户角色

（继承 MVP 角色定义，V1 扩展角色能力）

| 角色 | 描述 | V1 新增核心诉求 |
|------|------|-----------------|
| 租户管理员 | 注册账号的独立开发者或小团队负责人 | 管理 API Key、监控用量、评测 RAG 质量、预览组件效果、配置 LLM |
| 团队成员 | 被管理员邀请加入租户的协作者 | 配置评测集、查看用量统计、预览组件效果 |
| 终端访客 | 在嵌入的聊天组件中提问的最终用户 | （无变化）通过 API Key 鉴权访问对话服务 |

### 2.2 用户规模

- V1 阶段目标：200+ 注册租户
- 单租户 API Key 数量上限：5 个
- 单租户每日对话限额：管理员可配置，默认 1000 次
- 单租户评测集数量上限：10 个，每个评测集最多 50 个问答对

## 5. 非功能性需求

| 维度 | 要求 | 验收标准 |
|------|------|----------|
| 性能 | 对话 API < 2s（P95），管理后台 API < 500ms（P95） | 压测验证：100 并发下 P95 延迟满足要求 |
| 可用性 | 单实例部署，不要求高可用 | Docker Compose 部署，单节点可运行 |
| 安全性 | 多租户严格隔离 + JWT + API Key + 输入校验 + 文件上传校验 + 预览对话统计隔离 | 1. API Key 加密存储，展示脱敏 2. 预览对话不计入用量 3. postMessage 安全校验 origin |
| 数据量 | 单租户 API Key ≤ 5，评测集 ≤ 10×50，每日对话可配置限额 | 数据模型设计支持上述容量 |
| 国际化 | 中文界面，不做 i18n | UI 文案全中文 |
| 可测试性 | 核心 Service 单元测试覆盖率 > 80% | CI 中验证覆盖率 |
| 可观测性 | SLF4J + Logback，Request-Id 链路追踪，结构化日志，日志脱敏 | 每条日志包含 Request-Id，敏感信息不入日志 |
| 部署 | Docker Compose | 一键 `docker-compose up` 启动全部服务 |
| 兼容性 | V1 新增 Flyway 迁移脚本，不影响 MVP 已有数据 | 现有数据平滑迁移 |

## 6. 验收标准

### 6.1 V1 验收标准

- [ ] 管理员可生成 API Key，Key 生成时展示完整值仅一次，之后脱敏显示
- [ ] 管理员可吊销 API Key，吊销后使用该 Key 的请求立即返回 401
- [ ] 管理员可设置每日调用次数限制，超出后返回 429
- [ ] 聊天组件使用 API Key 鉴权，替代 MVP 阶段的简单 tenantId token
- [ ] 用量统计面板展示近 7/30 天调用量趋势图、Token 消耗、对话数
- [ ] 预览对话不计入用量统计（AOP 切面根据鉴权类型区分）
- [ ] 管理员可创建评测集（最多 10 个），添加问答对（每集最多 50 对）
- [ ] 管理员可一键运行评测，查看 Hit Rate 和详细结果
- [ ] 评测结果保留历史，支持对比
- [ ] 管理后台组件配置页内嵌 iframe 预览窗口，外观配置实时反映
- [ ] 预览窗口可发起真实 RAG 对话，走与正式访客相同的后端代码路径
- [ ] 预览窗口刷新按钮可一键重置对话
- [ ] 管理员可配置 LLM API 端点 URL 和密钥
- [ ] LLM API 密钥加密存储，展示时脱敏
- [ ] 配置 LLM 后可测试连通性
- [ ] 未配置 LLM 时使用系统默认
- [ ] Flyway 迁移脚本不影响 MVP 已有数据
- [ ] 核心 Service 单元测试覆盖率 > 80%

## 7. 功能边界（不做）

| 不做项 | 原因 | 可能的V2规划 |
|--------|------|-------------|
| 实时计费/账单/支付集成 | V1 只做调用次数硬限制，不做金额计算和支付 | V2 考虑 |
| 多语言 UI | 目标用户是独立开发者/小团队，中文界面即可 | V2 按需 |
| 对话历史持久化与导出 | 终端访客的对话是临时性的，问完即焚 | V2 按需 |
| 多知识库/多 Bot | 每个租户一个知识库一个 Bot，不做多实例 | V2 按需 |
| SSO/OAuth 第三方登录 | 目标用户是小团队，邮箱注册即可 | V2 考虑 |
| 文档协作编辑 | 产品定位是"上传+生成"，不是文档编辑器 | 不计划 |
| Webhook/事件回调 | V1 无集成需求 | V2 按需 |
| 移动端适配（管理后台） | 管理后台桌面端优先 | V2 按需 |
| API Key 权限分级 | V1 所有 Key 权限相同，不做读写分离 | V2 考虑 |
| 用量统计导出 | V1 只在管理后台展示，不支持 CSV/Excel 导出 | V2 考虑 |
| 评测集自动定时运行 | V1 只支持手动触发评测 | V2 考虑 |
| 评测集 NLP 评测指标 | V1 只做检索 Hit Rate，不做 BLEU/ROUGE 等生成质量指标 | V2 考虑 |

## 8. 依赖与约束

### 8.1 外部依赖

| 依赖项 | 类型 | 影响 |
|--------|------|------|
| LLM API（租户自配或系统默认） | 技术 | 对话生成依赖此服务，服务不可用时对话功能不可用 |
| PostgreSQL 16+ | 技术 | 主数据存储，新增 API Key/统计/评测表 |
| Milvus 2.4+ | 技术 | 向量检索依赖，评测功能依赖 |
| Redis 7.0+ | 技术 | 缓存 + 异步任务队列 + 每日调用计数 |
| Docker + Docker Compose | 技术 | 部署依赖 |

### 8.2 约束条件

- 必须使用 Java 21 + Spring Boot 3.3+ + Vue 3 技术栈（见 `.ai/tech.md`）
- 代码必须遵循 `.ai/codeRule.md` 规范
- 项目结构必须遵循 `.ai/structure.md` 规范（V1 新增 module-apikey、module-stat、module-eval）
- V1 预览对话遵循"同路径原则"（见 `.ai/structure.md` V1 预览对话架构规则）
- 单实例 Docker Compose 部署
- 中文界面，不做国际化
- 🚫 禁止向 LLM 提交图片

### 8.3 MVP 兼容性约束

- V1 的 Flyway 迁移脚本从 V5 开始（V1-V4 已被 MVP 使用）
- 现有 API 接口保持兼容，不删除已有端点
- ChatController 的 `/api/v1/chat/conversations` 端点同时接受 JWT（预览）和 API Key（正式）两种鉴权
- Widget 嵌入代码需从 widget_token 迁移到 API Key，但保持向后兼容过渡期
